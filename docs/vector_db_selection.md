# Greenvest Agent — Vector Database Selection

## What This System Actually Needs

Before evaluating options, pin down the hard requirements. The vector DB must satisfy all of these:

| Requirement | Why |
|---|---|
| Hybrid search (BM25 + dense ANN) with server-side RRF | Core retrieval mechanism for Branch B — semantic + keyword must fuse before results return |
| Numeric pre-filtering integrated into ANN traversal | `temp_rating_f <= 15`, `r_value >= 4.5` must filter *during* graph traversal, not after — post-hoc filtering destroys recall on selective conditions |
| ≤ 200ms P95 for Branch B at 500 concurrent sessions | Hard ceiling from `solution.md §2` |
| Configurable fusion weights | `w_dense=0.6/w_sparse=0.4` for feature queries; `w_dense=0.2/w_sparse=0.8` for brand/SKU queries |
| Self-hosted option | Data residency, cost at scale, no vendor lock-in on critical retrieval path |
| Async Python client | Every retrieval node uses `asyncio.gather` |

---

## The Five Contenders

### 1. Weaviate

Hybrid search is native and turnkey. Weaviate maintains an inverted BM25F index internally — no external sparse vector generation required. You pass a query string; Weaviate runs BM25F and dense ANN in parallel and fuses them server-side.

**Fusion:** Two modes available:
- `rankedFusion` — RRF using rank positions (rank-based, the original default)
- `relativeScoreFusion` — uses actual normalized scores rather than rank positions (default since v1.24, required for `autocut`)

The `alpha` parameter (0–1) is the weighting knob. `alpha=0.6` means 60% dense, 40% BM25. It maps directly to the spec's `w_dense`.

**Pre-filtering:** The `acorn` filter strategy (default since v1.34) integrates filters *during* HNSW graph traversal rather than post-hoc. This is the right architecture for selective numeric filters. Pre-v1.34 Weaviate had the post-hoc problem; make sure you're on v1.34+.

**Auto-embedding:** The `text2vec-openai` module handles query embedding server-side using the configured model and dimension. This is what eliminates the vector space mismatch problem described in `embedding_strategy.md` — Weaviate guarantees query and document use the same model.

**Weaknesses:**
- BM25F is less configurable than Elasticsearch — you cannot tune `k1` and `b` per field. For a gear catalog where brand token exact-match matters a lot, this is a real gap.
- `alpha` is a single coarse weight. You can't independently control the candidate list sizes for each branch before fusion.
- Resource-hungry relative to Qdrant — JVM-based core.
- Younger company than Elasticsearch/OpenSearch; longevity risk is real.

---

### 2. Qdrant

Qdrant's hybrid search architecture is different from Weaviate's and is worth understanding correctly: it does **not** have a built-in BM25 inverted index. Instead, it stores both sparse and dense vectors as named, co-located fields on each point. Hybrid search fuses two *vector* searches — one sparse, one dense — server-side.

The sparse vectors come from a model like **SPLADE** or **miniCOIL** (via Qdrant's FastEmbed library). This is not a limitation — it's actually better. SPLADE significantly outperforms classical BM25 on retrieval benchmarks (MRR@10 of 0.34 vs BM25's 0.18) because it performs term expansion: a query for "warm sleeping bag" will produce sparse vector activations for related terms like "insulated", "synthetic fill", "cold weather" that appear in the vocabulary. BM25 doesn't do this.

**Fusion:** Three server-side methods:
- **RRF** — rank-position based, tunable `k` constant
- **Weighted RRF** — assign independent weights per prefetch branch: `Rrf(weights=[2.0, 1.0])` favors dense. This maps cleanly to the spec's dynamic weight switching between feature queries and brand queries.
- **DBSF** (Distribution-Based Score Fusion) — normalizes score distributions before summing; best when you have characterized score distributions per branch

The `prefetch` API is architecturally cleaner than Weaviate's `alpha`: you express each branch of the hybrid search independently, with its own filter, limit, and vector field. The fusion step is separate from the retrieval step.

```python
results = client.query_points(
    collection_name="rei_products",
    prefetch=[
        # Branch: sparse (SPLADE)
        models.Prefetch(
            query=sparse_vector,   # generated by FastEmbed
            using="sparse_text",
            limit=50,
            filter=models.Filter(must=[
                models.FieldCondition(key="temp_rating_f", range=models.Range(lte=15))
            ])
        ),
        # Branch: dense
        models.Prefetch(
            query=dense_vector,    # text-embedding-3-large @ 1536
            using="dense_text",
            limit=50,
            filter=models.Filter(must=[
                models.FieldCondition(key="temp_rating_f", range=models.Range(lte=15))
            ])
        ),
    ],
    query=models.RrfQuery(rrf=models.Rrf(weights=[1.0, 2.0])),  # favor dense
    limit=5,
)
```

**Pre-filtering:** Qdrant's query planner switches between HNSW graph traversal and brute force based on filter selectivity. For a filter returning 10% of the collection, it traverses the graph with filter-aware expansion. For a filter returning 1%, it falls back to brute force over the filtered set. This adaptive behavior handles the full range of filter selectivity without recall degradation.

**Weaknesses:**
- You must generate and store sparse vectors at index time. If you switch sparse models, you re-index.
- Sparse model inference at query time adds ~5–20ms depending on model size and hardware.
- More moving parts than Weaviate's turnkey approach.

### 3. Pinecone

The 100 QPS cap on sparse indexes is the deal-breaker for this system. At 500 concurrent sessions this cap is hit immediately. Beyond that:

- No true RRF — fusion is client-side linear interpolation (`alpha × dense + (1-alpha) × sparse`), not rank-based. Linear interpolation requires score calibration; dense and sparse scores are on different scales. RRF doesn't have this problem because it only uses rank positions.
- No self-hosted option. All data lives in Pinecone's infrastructure or BYOC at enterprise pricing.
- Pre-filtering is post-hoc (applied after ANN), which degrades recall under selective filters.

**Verdict: ruled out.** The QPS cap, fusion quality, and self-hosted requirement together make Pinecone the wrong choice for this system.

---

### 4. pgvector (PostgreSQL Extension)

pgvector is compelling specifically because **PostgreSQL is already in the stack** for Branch C inventory queries. Collapsing Branch B retrieval into the same PostgreSQL instance would mean one fewer infrastructure component, shared connection pooling, and SQL-native joins between catalog results and inventory data in a single query.

**Hybrid search via SQL composition:**

```sql
WITH dense_results AS (
    SELECT id, ROW_NUMBER() OVER (ORDER BY embedding <=> $1) AS rank
    FROM products
    WHERE temp_rating_f <= 15 AND r_value >= 4.5
    ORDER BY embedding <=> $1
    LIMIT 50
),
bm25_results AS (
    SELECT id, ROW_NUMBER() OVER (ORDER BY ts_rank(search_vec, $2) DESC) AS rank
    FROM products
    WHERE temp_rating_f <= 15 AND r_value >= 4.5
      AND search_vec @@ $2
    LIMIT 50
)
SELECT COALESCE(d.id, b.id) AS id,
       1.0/(60 + COALESCE(d.rank, 100)) + 1.0/(60 + COALESCE(b.rank, 100)) AS rrf_score
FROM dense_results d
FULL OUTER JOIN bm25_results b ON d.id = b.id
ORDER BY rrf_score DESC
LIMIT 5;
```

This is real RRF in SQL. It works. The problems:

- PostgreSQL's `ts_rank` is a BM25 *approximation*, not true BM25. It uses TF-IDF-based scoring without BM25's document length normalization. For a product catalog with short, fairly uniform descriptions, this gap is small — but it's there.
- Two sequential index scans + a FULL OUTER JOIN per query. No single-pass execution.
- At 500 concurrent sessions, you need PgBouncer for connection pooling. Even with pooling, the combined Branch B + Branch C load on the same PostgreSQL instance is a risk — a slow catalog query competes with inventory reads for connections.
- HNSW build times are slow for large datasets and need explicit parallelism tuning.

**ParadeDB** (`pg_search` extension) fixes the BM25 gap by adding a true BM25 index via the Tantivy Rust library, and it's designed specifically to compose with pgvector for hybrid search. It's actively developed but not yet at the operational maturity of purpose-built databases (as of early 2026).

**Verdict: viable as a simplicity trade-off, not as a performance-first choice.** If the team's priority is fewer infrastructure components and SQL fluency, pgvector + ParadeDB is a legitimate path. If retrieval quality is the priority, a dedicated vector DB wins.

### 5. Elasticsearch / OpenSearch

Elasticsearch has the best BM25 implementation of any option here — it is what Elasticsearch was built on. BM25 parameters (`k1`, `b`) are configurable per field. For a product catalog where exact keyword match matters for brand names, material codes (GORE-TEX, PrimaLoft, 800-fill), and SKUs, this is a meaningful advantage.

**Fusion via RRF (Elasticsearch 8.x+):**

```json
{
  "retriever": {
    "rrf": {
      "retrievers": [
        {
          "standard": {
            "query": { "multi_match": { "query": "insulated sleeping bag", "fields": ["name^2", "description"] } }
          }
        },
        {
          "knn": {
            "field": "embedding",
            "query_vector": [...],
            "num_candidates": 100,
            "filter": { "range": { "temp_rating_f": { "lte": 15 } } }
          }
        }
      ],
      "rank_window_size": 100,
      "rank_constant": 60
    }
  }
}
```

`rank_constant: 60` is the `k` in RRF — exactly what the spec calls for.

**In-graph filtering:** Elasticsearch applies the `filter` during HNSW kNN traversal, not after. This is the same correct architecture as Weaviate's `acorn` and Qdrant's adaptive planner.

**OpenSearch** covers the same ground via its `hybrid` query type + normalization-processor pipeline. The pipeline approach (normalization-processor with min-max or L2 normalization + weighted scoring) is more manual to configure than Elasticsearch's inline RRF but achieves equivalent results. AWS OpenSearch Service is 20–40% cheaper than Elastic Cloud for equivalent workloads.

**Weaknesses:**
- JVM operational complexity: heap sizing, GC tuning, shard configuration, index lifecycle management. This is a non-trivial ops investment.
- All vector data must fit in the node's page cache for low latency. Memory planning is critical.
- RRF in Elasticsearch does not support scroll or sort — pagination has constraints in hybrid mode.
- OpenSearch's normalization-processor requires a separate search pipeline configuration step.
- Heaviest infrastructure footprint of all options.

**Verdict: right choice if you already operate Elasticsearch/OpenSearch at scale.** For a greenfield build where the team doesn't already have Elasticsearch expertise, the operational overhead is hard to justify when Qdrant or Weaviate achieve similar retrieval quality with far less ops work.

## The Verdict

### Primary recommendation: **Qdrant**

Qdrant wins on three dimensions that matter most for this system:

**1. Retrieval quality.** SPLADE sparse vectors outperform classical BM25 (MRR@10 of 0.34 vs 0.18) because they do term expansion — "I sleep cold" expands to activations for "insulated", "low temperature rating", "warmth". BM25 doesn't do this. For a gear catalog where users describe products subjectively, this matters more than having BM25 built-in.

**2. Fusion control.** Qdrant's `prefetch` + `Rrf(weights=[...])` pattern maps exactly to the spec's dynamic weight switching. Switching from `w_dense=0.6` (feature queries) to `w_dense=0.2` (brand queries) is two lines of code changing the weights array. In Weaviate you change a single `alpha` — less expressive.

**3. Performance.** Qdrant benchmarks consistently show the lowest latencies and highest throughput for filtered ANN search. At 500 concurrent sessions, this margin matters.

The trade-off is managing the sparse vector pipeline: you run FastEmbed (or SPLADE) at index time to generate sparse vectors alongside dense vectors, and at query time to encode the enriched query. This adds ~5–20ms and a dependency on FastEmbed. This is the price of the quality advantage.

### When to choose Weaviate instead

Choose Weaviate if:
- The team wants a faster path to Phase 11 with less pipeline engineering
- You want the auto-embedding modules to guarantee model/dimension consistency at the infrastructure level rather than in code
- The `text2vec-openai` module's automatic query embedding is worth the reduced fusion control

Weaviate is not a bad choice — it is turnkey hybrid search that works well. Qdrant is a better technical fit for this specific system's requirements.

### When to choose pgvector instead

Choose pgvector if:
- Minimizing infrastructure components is the top priority over retrieval quality
- The team has strong SQL/PostgreSQL expertise and limited vector DB experience
- You want Branch B catalog queries and Branch C inventory queries in a single database, enabling SQL joins between catalog results and live inventory in one round trip

The Branch B + Branch C consolidation argument is genuinely compelling. A single `SELECT` that does hybrid search on the catalog and joins inventory availability is simpler than `asyncio.gather` across two systems. The retrieval quality trade-off is real but manageable for a catalog of REI's scale.

### Don't choose Elasticsearch unless

- The team already operates Elasticsearch at scale
- Exact BM25 keyword match precision for brand/SKU/material queries is demonstrably failing with other options
- You have dedicated infrastructure ops capacity

### Ruled out: Pinecone

The 100 QPS cap on sparse indexes is a hard wall at 500 concurrent sessions. No self-hosted option. Linear interpolation fusion is architecturally inferior to RRF for this use case. Don't build on it.

## Migration Path

Whichever database you choose at Phase 11, the `search_catalog(state)` interface in `branch_b_catalog.py` is unchanged. The migration path between databases is contained to that one file.

**Suggested progression:**
1. **Phase 11:** Ship with Weaviate Cloud (managed, lowest ops overhead, fastest to production)
2. **After Phase 18 (eval data):** Use the labeled query → product pairs to benchmark Qdrant against Weaviate on real traffic
3. **Migrate to Qdrant** if Recall@5 or MRR improvements justify it — the interface contract makes this a single-file swap

This keeps Phase 11 launch risk low while leaving the door open for a quality-driven migration once you have real eval data to justify it.

## If You Go With Qdrant — Sparse Vector Pipeline

The only meaningful extra work vs. Weaviate is the sparse vector pipeline.

**At index time:**
```python
from fastembed import SparseTextEmbedding

sparse_model = SparseTextEmbedding(model_name="prithivida/Splade_PP_en_v1")

def build_point(product: dict) -> PointStruct:
    doc = build_product_document(product)
    dense_vec = embed_dense(doc)           # text-embedding-3-large @ 1536
    sparse_vec = list(sparse_model.embed([doc]))[0]

    return PointStruct(
        id=product["sku"],
        vector={
            "dense_text": dense_vec,
            "sparse_text": SparseVector(
                indices=sparse_vec.indices.tolist(),
                values=sparse_vec.values.tolist(),
            ),
        },
        payload={k: v for k, v in product.items() if k != "sku"},
    )
```

**At query time:**
```python
async def search_catalog(state: GreenvestState) -> list[dict]:
    query_doc = build_query_document(state)

    dense_vec = await embed_dense_async(query_doc)
    sparse_vec = list(sparse_model.embed([query_doc]))[0]

    has_brand = _detect_brand_token(state["query"])
    dense_weight, sparse_weight = (0.2, 1.0) if has_brand else (2.0, 1.0)

    spec_filter = _build_qdrant_filter(state.get("derived_specs", []))

    results = client.query_points(
        collection_name="rei_products",
        prefetch=[
            models.Prefetch(
                query=SparseVector(indices=sparse_vec.indices.tolist(), values=sparse_vec.values.tolist()),
                using="sparse_text",
                filter=spec_filter,
                limit=50,
            ),
            models.Prefetch(
                query=dense_vec,
                using="dense_text",
                filter=spec_filter,
                limit=50,
            ),
        ],
        query=models.RrfQuery(rrf=models.Rrf(weights=[sparse_weight, dense_weight])),
        limit=5,
    )

    return [hit.payload for hit in results.points]
```

The SPLADE inference (`sparse_model.embed`) adds ~8–15ms using FastEmbed's optimised ONNX runtime. At a 200ms Branch B ceiling, this is well within budget.

## Collection Schema (Qdrant)

```python
client.create_collection(
    collection_name="rei_products",
    vectors_config={
        "dense_text": models.VectorParams(
            size=1536,
            distance=models.Distance.COSINE,
            hnsw_config=models.HnswConfigDiff(m=16, ef_construct=100),
        ),
    },
    sparse_vectors_config={
        "sparse_text": models.SparseVectorParams(
            index=models.SparseIndexParams(on_disk=False),  # keep in RAM for latency
        ),
    },
)

# Payload indexes for pre-filtering
client.create_payload_index("rei_products", "temp_rating_f", models.PayloadSchemaType.FLOAT)
client.create_payload_index("rei_products", "r_value",       models.PayloadSchemaType.FLOAT)
client.create_payload_index("rei_products", "category",      models.PayloadSchemaType.KEYWORD)
client.create_payload_index("rei_products", "fill_type",     models.PayloadSchemaType.KEYWORD)
```

Payload indexes on `temp_rating_f` and `r_value` are critical — without them, Qdrant falls back to brute-force payload scanning for filter evaluation, which defeats the pre-filtering architecture.
